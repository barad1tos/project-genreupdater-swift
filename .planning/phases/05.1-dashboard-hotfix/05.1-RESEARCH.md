# Phase 5.1: Dashboard Hotfix - Research

**Researched:** 2026-02-23
**Domain:** SwiftUI loading state machine, shimmer animations, design token consistency
**Confidence:** HIGH

## Summary

This phase fixes two integration bugs identified by the v1.0 milestone audit and performs a broad tech debt cleanup across SharedUI and App. The bugs are well-understood with clear root causes in existing code. Bug 1: `DashboardView.swift:45-47` fires `.task(id: tracks.count)` on initial render with empty tracks, causing `refreshFromLive(tracks: [])` to set `.emptyLibrary` state before MusicKit data arrives -- a flash of "Your Music Library is Empty." Bug 2: `ShimmerPlaceholder(shape: .rectangle(width: .infinity, height: 44))` passes `CGFloat.infinity` as a fixed frame width, producing undefined layout behavior.

The tech debt scope is broader than just these two bugs. CONTEXT.md specifies a "broad cleanup across SharedUI + App" covering: raw Motion/Spacing literals replaced with tokens, stale comments removed, the orphaned `motionAnimation` extension preserved (per user decision), and a Periphery scan for dead code. The shimmer-to-content transition also needs enhancement with crossfade animation, minimum shimmer time, and stagger effects per CONTEXT.md decisions.

**Primary recommendation:** Fix the two integration bugs first (state machine guard + shimmer layout), then implement the enhanced shimmer transition (crossfade + minimum hold + stagger), then sweep all raw literals and dead code in a final tech debt pass.

## Architecture Patterns

### Current DashboardView Data Flow (before fix)
```
MainView.loadTracks()
  -> isLoading = true
  -> loadCachedSnapshot()                    # sets metricsSnapshot
  -> reader.fetchAllTracks()                 # async, takes seconds
  -> isLoading = false, tracks = [fetched]

DashboardView receives:
  tracks: [Track]          # starts as []
  metricsSnapshot: ...?    # nil on first launch
  isLoadingTracks: Bool    # true during fetch

  .onAppear -> loadCachedMetrics(nil) -> loadingState = .shimmer   (correct)
  .task(id: tracks.count)  -> tracks.count=0 -> refreshFromLive([])
                           -> loadingState = .emptyLibrary          (BUG!)
```

### Fixed Data Flow (after fix)
```
DashboardView:
  .task(id: tracks.count) {
      guard !isLoadingTracks || !tracks.isEmpty else { return }
      viewModel.refreshFromLive(tracks: tracks)
  }

Timeline:
  1. onAppear: loadCachedMetrics(nil) -> .shimmer
  2. task fires (count=0, isLoading=true) -> guard exits, stays .shimmer
  3. MainView finishes loading -> tracks=[38K], isLoading=false
  4. task fires again (count=38085) -> refreshFromLive -> .live
```

### Enhanced Shimmer Transition Pattern
```
DashboardViewModel state machine:

enum DashboardLoadingState {
    case shimmer           // First launch, no cache
    case cached(Date)      // Has cached data
    case updating          // Background refresh in progress
    case live              // Fresh data displayed
    case error(String)     // Failure
    case permissionDenied  // MusicKit denied
    case emptyLibrary      // Authorized + loaded + 0 tracks
}

Transition rules:
  shimmer -> live:           0.5s crossfade, minimum 0.5s shimmer hold
  shimmer -> error:          instant (no fade)
  shimmer -> permissionDenied: instant (no fade)
  shimmer -> emptyLibrary:   BLOCKED while isLoadingTracks is true
  cached -> live:            smooth crossfade (content update in place)
  any -> error:              instant
```

### Stagger Animation Pattern
```swift
// Three groups with 50ms stagger delay
VStack(spacing: 0) {
    gaugeSection
        .opacity(showGauge ? 1 : 0)

    metricsSection
        .opacity(showMetrics ? 1 : 0)

    quickActionsSection
        .opacity(showActions ? 1 : 0)
}
.onAppear {
    // Stagger: Hero first, then cards, then actions
    withAnimation(.easeOut(duration: 0.5)) { showGauge = true }
    withAnimation(.easeOut(duration: 0.5).delay(0.05)) { showMetrics = true }
    withAnimation(.easeOut(duration: 0.5).delay(0.10)) { showActions = true }
}
```

### Minimum Shimmer Time Pattern
```swift
// Track when shimmer started to enforce minimum hold
private var shimmerStartTime: Date?

func transitionToLive(tracks: [Track]) {
    let elapsed = Date().timeIntervalSince(shimmerStartTime ?? .distantPast)
    let remainingHold = max(0, 0.5 - elapsed)

    // Delay transition if shimmer hasn't been visible long enough
    Task {
        if remainingHold > 0 {
            try? await Task.sleep(for: .seconds(remainingHold))
        }
        // Now transition with crossfade
        withAnimation(.easeInOut(duration: 0.5)) {
            loadingState = .live
            metrics = computedMetrics
        }
    }
}
```

### Anti-Patterns to Avoid
- **Using `CGFloat.infinity` as a fixed frame width:** SwiftUI treats `.infinity` as "unconstrained" in layout proposals, not as "fill parent." Use `.frame(maxWidth: .infinity)` or the `.card` shape variant instead.
- **Checking `tracks.isEmpty` without considering loading state:** The empty check must be gated on loading completion. `tracks.isEmpty && !isLoading` is the safe pattern.
- **Using `.transition(.opacity)` on the container:** Crossfade should use `.opacity` on the content, not SwiftUI's transition system, to avoid layout jumps during state changes.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Shimmer animation | Custom gradient animation | SwiftUI-Shimmer `.shimmering()` | Already a dependency, handles animation timing and gradient automatically |
| Layout measurement | GeometryReader for shimmer sizing | Hardcoded dimensions matching real components | Simpler, avoids GeometryReader layout overhead, zero layout shift guaranteed |
| Reduce motion detection | Manual UserDefaults check | `@Environment(\.accessibilityReduceMotion)` | System-provided, auto-updates |

## Common Pitfalls

### Pitfall 1: Task fires on initial render with stale count
**What goes wrong:** `.task(id: tracks.count)` fires immediately when `tracks = []` (count=0). With `isLoading=true`, this is a loading state, not an empty library.
**Why it happens:** SwiftUI `.task(id:)` fires on first appear AND on id change. Initial `[]` is indistinguishable from "genuinely 0 tracks" by count alone.
**How to avoid:** Guard the task body: `guard !isLoadingTracks || !tracks.isEmpty else { return }`.
**Warning signs:** Flash of "Your Music Library is Empty" on first launch.

### Pitfall 2: CGFloat.infinity in fixed frame
**What goes wrong:** `ShimmerPlaceholder(shape: .rectangle(width: .infinity, height: 44))` passes `.infinity` to the `ShimmerShape.rectangle` case, which applies it as `.frame(width: width)` -- a fixed width of infinity.
**Why it happens:** Developer intended "fill available width" but used the wrong API.
**How to avoid:** Use `.frame(maxWidth: .infinity)` on the outside, or add a dedicated `.row` or `.quickAction` shape case that uses `maxWidth` internally.
**Warning signs:** Shimmer rows that overflow or collapse to zero width.

### Pitfall 3: Animation crossfade jumping layout
**What goes wrong:** Using SwiftUI `.transition()` between shimmer and live content causes a layout jump because both views measure at different sizes.
**Why it happens:** SwiftUI's transition system removes the old view and inserts the new view, causing a remeasure.
**How to avoid:** Use opacity-based crossfade within the same container. Keep shimmer and live content in the same VStack structure so they share layout. Use `ZStack` with opacity if structures differ.
**Warning signs:** Content jumping or resizing during the shimmer-to-live transition.

### Pitfall 4: Stagger timing not respecting reduce motion
**What goes wrong:** Stagger animation plays even when user has "Reduce Motion" enabled.
**Why it happens:** `withAnimation` ignores `accessibilityReduceMotion` by default.
**How to avoid:** Check `reduceMotion` at call site. If true, show all groups immediately without stagger.
**Warning signs:** Accessibility audit failure.

### Pitfall 5: motionAnimation extension removal breaks future phases
**What goes wrong:** Deleting the `motionAnimation` extension breaks a public API that future phases may use.
**Why it happens:** Zero current call sites tempts removal.
**How to avoid:** CONTEXT.md explicitly says "preserve and move to appropriate module" -- keep it in DesignTokens.swift.
**Warning signs:** N/A (user decision locks this).

### Pitfall 6: Raw literal sweep too aggressive
**What goes wrong:** Replacing `spacing: 0` with a Spacing token, or replacing animation literals that intentionally differ from Motion tokens.
**Why it happens:** Mechanical "replace all raw numbers" without considering intent.
**How to avoid:** `spacing: 0` is semantic (no spacing) and should stay as `0`. Only replace literals that match existing token values or where a new token is warranted. Animation literals in `ProgressRing` (0.4s) and `MainView.updateColumnVisibility` (0.25s) may need new tokens.
**Warning signs:** Visual regressions from wrong token values.

## Open Questions

1. **Column visibility animation token**
   - What we know: MainView uses 0.25s for column visibility changes, no existing token matches
   - What's unclear: Whether 0.25s is the right duration or if 0.3s (curveDefault) would work
   - Recommendation: Create `Motion.curveLayout` at 0.25s -- layout responsiveness benefits from being slightly faster

2. **Spacing values 2pt and 6pt**
   - What we know: Several files use `spacing: 2` and `spacing: 6` which don't match any Spacing token (smallest is `xxs: 4`)
   - What's unclear: Whether to create `Spacing.micro: CGFloat = 2` or leave as raw literals
   - Recommendation: Leave as raw -- these are sub-token values used for tight element proximity (e.g., label+sublabel pairs). Creating tokens for every value defeats the purpose of a scale.

3. **StatCard vs MetricCard**
   - What we know: MetricCard reimplements StatCard's hover/press pattern. Milestone audit flagged StatCard as orphaned.
   - What's unclear: Whether to refactor MetricCard to compose StatCard
   - Recommendation: Leave as-is for this phase. MetricCard has different layout (icon, trend, delta) that StatCard doesn't support. StatCard will be used by future phases. Not dead code.
