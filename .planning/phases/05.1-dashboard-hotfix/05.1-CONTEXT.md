# Phase 5.1: Dashboard Hotfix - Context

**Gathered:** 2026-02-23
**Status:** Ready for planning

<domain>
## Phase Boundary

Fix two integration bugs (shimmer timing on first launch and ShimmerPlaceholder layout) and clean up tech debt in Dashboard/SharedUI. Shimmer must stay visible until real data arrives — no flash of empty state. Quick-action shimmer rows must render at full container width. Broad token/dead-code consistency pass across SharedUI + App.

</domain>

<decisions>
## Implementation Decisions

### Shimmer-to-content transition
- Fade crossfade transition — shimmer fades out, content fades in
- Duration: 0.5s crossfade animation
- Minimum shimmer time: 0.5s — even if cache returns data in 50ms, shimmer holds for at least half a second to avoid flicker
- HeroGauge shimmer: gauge itself shows shimmer effect (circle with shimmer), not an empty 0% gauge
- Stagger: Hero → MetricCards → QuickActions appear with 50ms stagger between groups
- Shimmer style: Ayu palette colors (bgSecondary/bgTertiary) — not default SwiftUI-Shimmer colors
- Dark mode: Ayu colors adapt automatically, no separate shimmer colors needed

### Error handling during shimmer
- MusicKit error (denied, timeout): instant transition to error view — no fade, immediately obvious something went wrong
- Loading timeout: 15s — if MusicKit hasn't responded in 15s, show error state with "Retry" button

### Re-shimmer behavior
- Shimmer is only for first load (no cache, cold start)
- Subsequent visits (tab switch, background return): show stale data and update in place — no re-shimmer
- Stale data disappears only when new data arrives (smooth crossfade to updated values)

### Loading state machine
- Explicit loading state in ViewModel: `.loading` → `.loaded` → `.error`
- Empty state ("Your Music Library is Empty") only shown when state is `.loaded` AND `tracks.count == 0`
- Never show empty state while in `.loading` — this is the core bug being fixed

### Authorization states
- MusicKit not yet authorized (first launch, no prompt answer): separate UI — "Grant access to Music" with a prompt-triggering action
- MusicKit denied: error state + "Open System Settings" button to change permissions
- These are distinct from "library is empty" — never conflate them

### True empty state
- When library is genuinely empty (authorized + loaded + 0 tracks): illustration + text "Add music in Apple Music" + "Open Music.app" button
- Friendly tone, actionable

### ShimmerPlaceholder layout
- Quick-action shimmer rows: fill container width (`.frame(maxWidth: .infinity)`) — matches real QuickActionButton layout
- Shimmer placeholder count: exact match to real button count — same number of placeholders as real quick-action buttons
- Shimmer placeholder height: exact match to real QuickActionButton height — zero layout shift
- MetricCards shimmer: exact dimensions matching real MetricCard components — zero layout shift
- Corner radius: exact radius from DesignTokens (Radius.card for cards, Radius.button for actions) — pixel-perfect match
- Spacing between shimmer elements: matches real layout spacing

### Tech debt scope
- Broad cleanup across SharedUI + App — not limited to 3 success criteria items
- Categories: stale comments, raw literals, unused extensions/dead code, AND consistency (all Motion tokens, all Spacing tokens, all colors through Ayu)
- motionAnimation extension: if no call sites, preserve and move to appropriate module (not delete) — it may be useful in future phases
- Raw literals: when no existing DesignToken fits, create new tokens in DesignTokens — full consistency over simplicity
- Periphery scan: run on SharedUI + App to find additional dead code
- Commit organization: logical subcategories (e.g., "shimmer fix", "token consistency", "dead code cleanup") — not microcommits, not one giant commit

### Claude's Discretion
- Exact Ayu color shades for shimmer gradient
- State machine implementation details (enum vs separate booleans)
- Periphery findings triage (which dead code to remove vs keep)
- Specific Motion token values for new transitions
- How to structure the shimmer placeholder views internally

</decisions>

<specifics>
## Specific Ideas

- Shimmer should feel like content "materializing" — the 0.5s fade gives weight to the transition
- Stagger effect (Hero → Cards → Actions) adds a premium cascade feel, like Linear or Apple's own loading patterns
- Error transitions should be instant/jarring on purpose — signals that something is wrong, not a normal loading flow
- Zero layout shift is the goal: shimmer skeleton matches real layout pixel-for-pixel (size, spacing, corner radius)

</specifics>
